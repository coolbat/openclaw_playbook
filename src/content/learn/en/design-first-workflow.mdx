## Small enough means debuggable

The first workflow should not impress anyone. It should teach you how the system behaves. That means it needs a clear trigger, a limited task boundary, and an output you can evaluate in seconds.

## A useful workflow frame

- one trigger
- one transformation
- one destination

For example: collect a small set of links, summarize them, and post the summary into one internal channel.

## What to avoid

- multiple destinations
- multiple approval paths
- hidden background dependencies
- outputs that are hard to evaluate

> **If failure feels vague**
>   The workflow is probably too large. Cut scope before you add more debugging.
