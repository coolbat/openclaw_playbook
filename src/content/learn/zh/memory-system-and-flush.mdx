和 OpenClaw 聊久了，AI 会开始"忘事"——一小时前告诉它的内容，它好像没印象了。这不是 bug，是上下文压缩（compaction）在正常工作。问题在于，默认行为会丢掉你本来想保留的细节。

这篇指南覆盖三件事：防止重要信息在压缩时丢失、写出真正能被检索到的日志、以及接入语义搜索让 AI 不用读完所有文件就能找到旧上下文。

## 为什么 AI 会"失忆"

每个模型都有上下文窗口限制。当对话接近这个限制时，OpenClaw 会自动把旧消息压缩成摘要来腾出空间。压缩是有损的——细节会被丢掉。

结果就是：你正在做一个任务，AI 一直在跟踪某个重要信息，压缩触发了，它把那条线索丢了。

## 方案一：memoryFlush——在压缩触发前先保存

`memoryFlush` 让 AI 在压缩运行**之前**把重要信息写入文件。在 `openclaw.json` 里加上：

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "reserveTokensFloor": 20000,
        "memoryFlush": {
          "enabled": true,
          "softThresholdTokens": 4000
        }
      }
    }
  }
}
```

| 参数 | 含义 | 推荐值 |
|------|------|--------|
| `reserveTokensFloor` | 压缩后至少保留多少 token 的最近对话 | `20000` |
| `memoryFlush.enabled` | 是否开启压缩前自动保存 | `true` |
| `memoryFlush.softThresholdTokens` | 距离压缩多少 token 时触发保存 | `4000` |

`memoryFlush` 静默执行，不打断对话。开了 verbose 模式（`/verbose`）可以看到 `🧹 Auto-compaction complete` 的提示。

对于关键会话，也可以手动触发并给出提示：`/compact keep technical decisions and current task state`。这样可以明确告诉 AI 要保留什么。

## 方案二：结构化日志——写出 memorySearch 能找到的记忆

`memorySearch` 底层用的是向量语义检索。检索质量几乎完全取决于日志条目怎么写。非结构化的散文命中率低，带清晰标签和标签的结构化条目命中率高。

### 有效的日志格式

```markdown
### [PROJECT:名称] 标题

- **结论**: 一句话总结结果
- **文件变更**: 涉及的文件路径
- **教训**: 踩坑点（如有）
- **标签**: #tag1 #tag2 #tag3
```

### 为什么这个格式有效

- `[PROJECT:名称]` 前缀让 AI 不用读所有内容就能按项目过滤
- **结论** 是语义搜索最先匹配的字段——写得越具体越好
- **标签** 是检索的显式锚点——在不同条目里保持一致的词汇
- 条目短意味着检索时能在上下文里放更多条目

### 对比示例

搜索词："nginx 部署问题"

命中率低——非结构化：
```
今天搞了服务器。试了几种 nginx 配置，遇到了一些问题，最后改了 upstream
设置之后搞定了……
```

命中率高——结构化：
```markdown
### [PROJECT:MyApp] nginx 反代配置完成
- **结论**: nginx 反代，监听 80 端口，upstream 127.0.0.1:3000
- **文件变更**: `/etc/nginx/sites-available/myapp`
- **教训**: upstream 用 127.0.0.1 不要用 localhost（IPv6 解析问题）
- **标签**: #myapp #nginx #deploy #upstream
```

结构化版本在你后来问"MyApp 的 nginx 问题是什么"时会稳定浮出来。非结构化版本可能找不到。

## 方案三：接入免费 embedding API 开启 memorySearch

默认情况下 `memorySearch` 可能没有配置。接入一个免费的 embedding 模型，对长期运行的配置来说效果提升明显。

国内用户推荐 SiliconFlow 的 `bge-m3` 模型——完全免费、不需要信用卡、中英文双语支持好：

```json
{
  "memorySearch": {
    "enabled": true,
    "provider": "openai",
    "remote": {
      "baseUrl": "https://api.siliconflow.cn/v1",
      "apiKey": "你的 SiliconFlow API key"
    },
    "model": "BAAI/bge-m3"
  }
}
```

获取 SiliconFlow API key：去 siliconflow.cn 注册 → 控制台 → 创建 API key。`bge-m3` 模型完全免费，没有用量限制。

为什么推荐 `bge-m3`：1024 维向量、中英文双语性能强、完全免费。对大多数 OpenClaw 记忆工作负载来说，这是最合适的默认选择。

## 方案四：自动记忆维护

随着时间推移，每日日志会越积越多。已完成的旧任务和过期条目开始污染搜索结果。在 `HEARTBEAT.md` 里加上每周维护任务：

```markdown
## 记忆维护（每周一次）

读取 `memory/heartbeat-state.json`，检查 `lastMemoryMaintenance` 字段。
如果距今 >= 7 天：
1. 读最近 7 天的 `memory/YYYY-MM-DD.md` 日志
2. 提炼有价值的信息到 projects.md 或 lessons.md
3. 把已完成的一次性任务压缩为一行结论
4. 删除过期或已被替代的条目
5. 更新 heartbeat-state.json 的 lastMemoryMaintenance 为今天
```

创建 `memory/heartbeat-state.json` 来记录上次维护时间：

```json
{
  "lastMemoryMaintenance": "2026-01-01"
}
```

AI 会在心跳检查时读取这个文件，到达间隔时自动执行维护。

## 整体架构

完整的记忆系统由四层协同工作：

1. **memoryFlush** — 防止压缩时丢失信息
2. **结构化日志格式** — 让条目可被检索
3. **memorySearch + bge-m3** — 跨所有日志的语义召回
4. **每周维护** — 长期保持搜索索引干净

这些都不需要大量配置时间。`memoryFlush` 配置是五行 JSON。日志格式是一个习惯。embedding API 配置十分钟搞定。维护任务是 `HEARTBEAT.md` 里的一段话。

> **最常见的错误** 是把记忆写成日记——长篇、叙事性、以过程为中心。这种风格写起来自然，但检索效果差。写结论，不写故事。AI 需要找到答案，不需要重温过程。
