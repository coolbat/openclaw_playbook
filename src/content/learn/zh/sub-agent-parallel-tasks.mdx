默认情况下，OpenClaw 是单线程的：做完一件事再做下一件。对简单工作流来说没问题。但对于有多个独立步骤的任务——同时调研多个话题、检查不同服务的状态、批量处理一组内容——单线程执行就成了瓶颈。

子 Agent 让主 Agent 可以把独立任务派发给并行运行的独立进程。每个子 Agent 有自己的会话，执行完任务后汇报结果。主 Agent 负责协调，子 Agent 负责干活。

## 子 Agent 到底是什么

子 Agent 不是另一个 AI，而是主 Agent 派生出来处理特定任务的独立 OpenClaw 会话。它无法访问主会话的对话历史——它只能看到主 Agent 给它的任务描述。

这种隔离既是优势也是约束。子 Agent 能并行运行，正是因为它们相互独立。但它们无法隐式共享上下文——它们需要知道的一切都必须写在任务描述里。

## 模型分级策略

子 Agent 最大的实际价值之一是控制成本。大多数任务不需要最强的模型。把任务路由到合适的档位，可以把 token 消耗降低 60–70%。

一个实用的三档配置：

| 档位 | 模型别名 | 适用场景 |
|------|----------|----------|
| 高 | `opus` | 复杂架构设计、多文件重构、深度推理 |
| 中 | `sonnet` | 写代码、写脚本、结构化信息处理（默认） |
| 低 | `haiku` | 简单文件操作、格式转换、搜索汇总 |

在 `openclaw.json` 里配置模型别名，这样可以用档位名称而不是完整模型 ID 来引用：

```json
{
  "models": {
    "your-provider/claude-opus-4-6": { "alias": "opus" },
    "your-provider/claude-sonnet-4-6": { "alias": "sonnet" },
    "your-provider/claude-haiku-4-5": { "alias": "haiku" }
  }
}
```

然后在 `AGENTS.md` 里记录这套策略，让主 Agent 一致地执行：

```markdown
## 子 Agent 模型策略

派发子 Agent 时，按任务复杂度选择模型档位：

| 档位 | 别名 | 适用 |
|------|------|------|
| 🔴 高 | opus | 复杂推理、架构设计、多文件工作 |
| 🟡 中 | sonnet | 写代码、脚本、结构化处理（默认） |
| 🟢 低 | haiku | 搜索、摘要、文件操作、格式转换 |

大多数日常任务是 haiku 级别。不确定时默认用 sonnet。
```

## 写出真正有效的任务描述

子 Agent 启动时没有任何上下文。它们无法在任务中途提问。它们的输出质量几乎完全取决于主 Agent 写的任务描述质量。

烂的任务描述产出烂的结果：

```
帮我审查一下代码的安全问题。
```

好的任务描述包含：目标、范围、约束和输出格式：

```markdown
## 任务：API 端点安全审查

### 目标
审查 /root/project/src/ 目录下所有 .js 文件的 API 安全问题。

### 关注点
1. SQL 注入风险
2. 未验证的用户输入
3. 硬编码的密钥或 token
4. 缺少权限检查的 API 端点

### 约束
- 只读不写，不要修改任何文件
- 忽略 node_modules/ 和 test/ 目录

### 输出格式
按严重程度分级（🔴 致命 / 🟡 重要 / 🟢 建议）。
每个问题给出：文件路径、行号、问题描述、修复建议。

### 结果
写入 /root/project/SECURITY-REVIEW.md
```

区别不在于长度，而在于具体程度。子 Agent 需要明确知道看什么、忽略什么、产出什么格式。

## 并发上限

同时跑太多子 Agent 会触发 API 限流。实际经验值是**最多同时跑 2 个子 Agent**。跑到 4 个基本必触发 429 错误。

这意味着在派发之前需要考虑任务依赖关系：

- **独立任务**（A 的输出不是 B 的输入）：并行运行，每批最多 2 个
- **有依赖的任务**（B 需要 A 的输出）：串行运行——等 A 完成后再派发 B

在 `AGENTS.md` 里记录这条规则：

```markdown
## 子 Agent 规则

- 同时最多运行 2 个子 Agent
- 派发并行任务前先检查依赖关系
- B 需要 A 输出的任务必须串行
- 不需要推理的任务用 haiku
```

## 一个实际的并行派发示例

假设你要：（1）汇总三个竞品网站，（2）检查自己的服务健康状态。这些任务相互独立——分两批并行派发：

```
第一批（并行）：
  - 子 Agent A（haiku）：抓取并汇总 competitor-1.com
  - 子 Agent B（haiku）：抓取并汇总 competitor-2.com

第二批（第一批完成后并行）：
  - 子 Agent C（haiku）：抓取并汇总 competitor-3.com
  - 子 Agent D（haiku）：检查服务健康端点并报告状态
```

主 Agent 收集四个结果，组装最终输出。总耗时大约是串行执行的一半。

## 什么时候不该用子 Agent

子 Agent 有协调开销。对简单、快速的任务来说，这个开销不值得：

- 单次网页抓取和摘要：直接在主会话里做
- 快速读文件然后回复：不需要派发
- 10 秒以内能完成的任务：串行就够了

子 Agent 在以下情况才值得用：任务真正相互独立、每个任务需要一定时间、可以并行运行且不需要彼此的输出。

> **主 Agent 的职责是协调，不是执行。** 一旦子 Agent 跑起来，不要让主 Agent 自己去做繁重的工作。它的价值在于路由、汇总结果、决定下一步——而不是亲自完成每一个任务。
